# Plan — #8 deploy-mode always resolves to 'unknown' — dead field probe against Coolify's application endpoints

- **Status:** ready (approved)
- **Type / priority / estimate:** bug · P0 · ~1h
- **blocked_by:** none

## Approach

`GET /applications/{uuid}` never eager-loads the `settings` relation upstream
(`ApplicationsController` never calls `->with('settings')`, and `Application`
has no `protected $with`), so `is_auto_deploy_enabled` — and the two flattened
guesses `.is_auto_deploy_enabled` / `.auto_deploy_enabled` — are never present
in the response. The three-way jq probe is therefore structurally guaranteed to
miss, and `deploy_mode` caches as `"unknown"` forever. A quick check for a
read-token-accessible endpoint exposing the flag (e.g.
`/applications/{uuid}/settings`) came up empty in the issue's upstream trace;
this plan takes the acceptance-safe path and does **not** assume one exists. If
the implementer discovers one during the fix, that is a scope change worth
raising with the product lead before proceeding — do not silently expand.

Concrete changes, in order:

1. **`cmd_deploy_mode` (scripts/atelier-coolify ~216–259).** Delete the
   three-way `is_auto_deploy_enabled` probe (lines ~243–247). The live-path jq
   now computes `deploy_mode` as the honest terminal state `"unknown"` (the API
   cannot tell auto from manual) and reports `auto_deploy_enabled: null`. Keep
   the fields that *do* serialize on the `Application` model and remain useful
   context: `uuid`, `name`, `git_repository`, `watched_branch`,
   `git_source_connected`. Add a short comment explaining *why* the flag is
   unreadable (pointing at the settings-relation root cause), so the next
   maintainer doesn't re-add the guess — this is exactly how commit `d6c3fd9`
   (M7.1.F54) introduced it.

2. **`cmd_status` (scripts/atelier-coolify ~164–175).** This carries a second,
   identical dead probe: the `auto_deploy` field (lines ~168–171) resolves to
   `null` for every real app for the same reason. Remove the probe. Recommended:
   drop the always-`null` `auto_deploy` key from the `status` output entirely
   (it is misleading, not merely empty) — see the open question below if the
   product lead prefers keeping an explicit `auto_deploy: null` for shape
   stability.

3. **Operator override = the existing cache file (no new mechanism).** The
   simplest override consistent with existing conventions already exists: a
   hand-edited `.coolify-deploy-mode.json` entry is returned verbatim by
   `_deploy_mode_cache_get` (with `source: "cache"`) and is never overwritten
   unless `--refresh` is passed. So an operator who knows an app is auto can set
   `{"deploy_mode":"auto"}` for that uuid and the skill will honor it. No new
   flag, no new code — just document this as the override. (`--refresh`
   re-fetches live and thus resets to `"unknown"`, which is correct behavior.)

4. **Graceful old-cache handling.** `_deploy_mode_cache_get` echoes the stored
   object with `jq -e '.[$u]'`, independent of the entry's internal shape, so
   pre-fix entries (which carry `deploy_mode:"unknown"`, `auto_deploy_enabled:null`)
   already read back without crashing. The fix must preserve this property:
   do not add any jq that hard-fails on a missing key when reading the cache.
   No cache migration or version stamp is warranted for a ~1h fix.

5. **Docs.** Update `skills/coolify/SKILL.md`, `README.md`, and `CLAUDE.md` (and
   the in-script `usage()` line ~70) to state that the auto-deploy flag cannot
   currently be read from the Coolify API, that the skill therefore always
   treats apps as **manual** (run an explicit `deploy` for code changes), and
   that the per-project cache file is the operator's override for known-auto
   apps. The "skip redundant deploy on auto-configured apps" optimization is
   effectively dormant until either the operator overrides via cache or an API
   surface for the flag appears — say so plainly rather than implying auto
   detection works.

6. **Version bump.** Bump `VERSION` in `scripts/atelier-coolify` (line 15) and
   `version` in `.claude-plugin/plugin.json`, keeping the two in lockstep as
   prior PRs did (#6 bumped both 0.3.0 → 0.4.0). See open question on
   patch-vs-minor.

## Acceptance criteria

- [ ] The three-way auto-deploy field probe is removed from `cmd_deploy_mode`;
      the live path no longer attempts to resolve `auto` vs `manual` from
      `.settings.is_auto_deploy_enabled` / `.is_auto_deploy_enabled` /
      `.auto_deploy_enabled`.
- [ ] The identical dead probe in `cmd_status` (`auto_deploy` field) is removed
      so `status` no longer emits a permanently-`null`/misleading value.
- [ ] `deploy-mode` reports a documented, honest terminal state for real apps
      (recommended: `deploy_mode: "unknown"` with `auto_deploy_enabled: null`)
      and the skill guidance treats apps as **manual** for the
      skip-redundant-deploy optimization unless the operator overrides.
- [ ] An operator override path is documented and works with existing code — a
      hand-edited `.coolify-deploy-mode.json` entry is honored by
      `deploy-mode` (source `cache`) and not clobbered except on `--refresh`.
- [ ] Existing `.coolify-deploy-mode.json` cache entries written before this fix
      read back without crashing (old-format entries degrade gracefully).
- [ ] `SKILL.md`, `README.md`, `CLAUDE.md`, and the in-script `usage()` text no
      longer claim the API exposes the auto-deploy flag; they state it cannot be
      read, apps are treated as manual, and describe the cache override.
- [ ] `VERSION` in `scripts/atelier-coolify` and `version` in
      `.claude-plugin/plugin.json` are bumped in lockstep, following HISTORY.md
      convention, with a matching HISTORY.md entry referencing the PR.
- [ ] Validation matches this repo's established bar (see HISTORY.md): `bash -n`
      and `shellcheck -S warning` clean; a hermetic scratch check that
      `deploy-mode` on a mocked `GET /applications/{uuid}` (no settings relation)
      yields the terminal state, that a hand-seeded cache entry is honored, and
      that a pre-fix-format cache entry does not crash.

## Decisions (product lead, on approval)

- **Terminal state: `unknown`**, not `manual`. Keep `deploy_mode: "unknown"` +
  `auto_deploy_enabled: null` on the live path — most honest, since the flag is
  genuinely unreadable. The skill's existing `unknown` → treat-as-manual
  mapping already gives the safe behavior; do not collapse the reported state
  to `"manual"` itself.
- **`cmd_status` output shape: drop the key.** Remove the always-`null`
  `auto_deploy` field from `status` output entirely rather than keeping an
  explicit `auto_deploy: null`. It's a thin internal CLI with no test/tooling
  depending on the field's presence.
- **Version bump: patch, `0.4.0` → `0.4.1`.** This removes dead/misleading
  behavior rather than adding a capability; #6's minor bump doesn't bind this
  case.
